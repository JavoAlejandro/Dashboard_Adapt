# Proposal: Fleet Percentiles & Period Comparison for the Temporal Sub-tab

## Intent

The Empresa tab's Temporal sub-tab shows only raw per-company averages of p/h and GSE/edad metrics — no benchmarking context. A company cannot answer its core question: **"In what period did variable X change for my fleet, and is that good or bad relative to everyone else?"** Two gaps drive this:

1. **No cross-company benchmarking** — a company sees its own mean but not where it sits in the fleet-wide distribution for the same period. No percentile logic exists for these metrics anywhere (the only percentile code, `ruido.js`, is hardcoded to dB and not reusable).
2. **No clean period-over-period self-comparison** — companies cannot place period A next to period B to see if results improved or worsened.

Separately, `hora_salida` currently has **outsized prominence** (a dedicated, always-rendered `_renderHoraChart` plus equal dimension weighting) that the user explicitly wants de-emphasized.

This builds additively on the completed `restructure-empresa-panel` refactor (`core.js` TOKENS + `fetchParseCsv`, `temporalIngest(rows)` interface) and must not alter it.

## Scope

### In Scope
- **New standalone Python precompute script** (separate from `Definitos/BASE_Im.py` and from `DATA/RUIDO/.../build_dashboard_kpis.py`, but mirroring the latter's `fleet_reference()` + `percentile_of()` pattern using `pandas.Series.quantile`). Reads the already-computed `DATA/IMPACTOS/impactos_*.csv`, outputs (a) a fleet-percentile **reference** CSV (p10/p25/p50/p75/p90 per metric per period) and (b) a **per-company percentile** CSV, broken out by period (`mes` at minimum). Output shape mirrors Ruido's two-file pattern.
- **Dashboard consumption**: fetch new reference CSV(s) via `core.js` `fetchParseCsv`, as a **separate additive fetch** (not routed through `temporalIngest`, which is per-company row ingest — this data is fleet-wide, different shape).
- **Benchmark visual** added to Temporal showing a company's metric-in-period vs the fleet percentile distribution.
- **Period-A-vs-period-B comparison UI** for the same company (self-comparison).
- **Remove `hora_salida` prominence** — `_renderHoraChart` (the dedicated always-on bar chart) is deleted entirely, per resolved decision.

### Out of Scope
- **R2 upload automation** — the reference CSV is generated by script and **manually uploaded to R2 by the user** (same manual process as all other data files; explicitly declined). Dashboard code never triggers upload or script runs.
- **Congestión / Emisión gauges** — explicitly deferred to a later, separate change per the user's priority order.
- Any change to `BASE_Im.py` or to the completed `restructure-empresa-panel` work.
- Re-running the full H3/GPS crossing — the existing `impactos_*.csv` outputs are the script input.

## Capabilities

### New Capabilities
- `temporal-fleet-benchmark`: cross-company percentile benchmarking of Temporal metrics against a precomputed fleet reference, per period.
- `temporal-period-comparison`: same-company period-A-vs-period-B self-comparison in the Temporal sub-tab.

### Modified Capabilities
- `temporal-subtab` (existing Temporal behavior): de-emphasis of `hora_salida` as a dimension/chart. *(No formal spec file exists yet; sdd-spec to establish or delta as appropriate.)*

## Approach

Two decoupled halves. **Offline**: a new argparse-driven Python script ingests `impactos_*.csv`, groups by company and period, computes fleet quantiles per metric per period plus each company's percentile rank, and emits two CSVs whose columns follow Ruido's `dashboard_kpis_referencia.csv` / `dashboard_kpis_empresa.csv` convention. The user runs it and uploads output to R2 manually. **In-browser**: Temporal fetches the reference CSV(s) additively via `fetchParseCsv`, joins them to the selected company/metric/period, and renders a benchmark visual + a period-vs-period comparison, reusing `TOKENS` colors. `_renderHoraChart` is deleted entirely. Load order and the dependency-free static architecture are preserved.

Unlike Ruido — whose reference file's `p10..p90` columns are **parsed but never used** (only the precomputed `_actual_pct` drives the needle) — this design should **actually consume the distribution** (e.g. a percentile band/marker), avoiding Ruido's dead-column pattern. Confirmed in question round.

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Domain/path name collides with the "Temporal" sub-tab label, causing confusion | Med | Use a distinct CSV domain prefix (proposed `flota/`, not `temporal/`); confirm in question round |
| Precompute column contract drifts from dashboard expectations | Med | Freeze the CSV schema in sdd-spec before implementing either half; mirror Ruido's proven shape |
| Percentile granularity chosen wrong (per-`mes` only vs `mes`×`dia_semana`) forces script rewrite | Med | Resolve granularity fork in question round before design |
| Gauge visual may not fit a time-series-oriented tab | Med | Treat visual as an open design decision; do not presuppose the gauge |
| Manual upload step forgotten → stale/missing reference file | Low | Dashboard degrades gracefully when reference CSV is absent (benchmark hidden, raw view intact) |
| No automated tests | High | Manual in-browser verification checklist (per project convention) |

## Rollback Plan

Dashboard changes are file-scoped on one branch — revert the branch/commits to restore prior Temporal behavior; no persisted state or schema migration. The reference CSV is additive data in R2; deleting/not-uploading it leaves the dashboard in its graceful-degradation (raw) state. The Python script is a standalone new file with no callers — removing it affects nothing in `dashboard_r2`.

## Dependencies

- Already-computed `DATA/IMPACTOS/impactos_*.csv` (present locally) as script input.
- `pandas` for the precompute script (offline only; not a dashboard dependency).
- `core.js` `fetchParseCsv` + `TOKENS` (from completed refactor) — unchanged.
- Papa Parse / Chart.js (already vendored) — unchanged.
- Resolution of the question-round decisions below before sdd-design.

## Success Criteria

- [ ] A company can see its metric-in-period positioned against the fleet percentile distribution for the same period.
- [ ] A company can compare itself across two periods (A vs B) and read whether it improved or worsened.
- [ ] `hora_salida` no longer has outsized prominence in the Temporal sub-tab.
- [ ] New reference CSV(s) consumed via `fetchParseCsv`, additively (not through `temporalIngest`); dashboard degrades gracefully when absent.
- [ ] The precompute script runs standalone against `impactos_*.csv` and emits the frozen-schema CSVs; verified in-browser after manual R2 upload.

## Proposal question round — RESOLVED

Decisions confirmed by user on 2026-07-14:

1. **Percentile granularity**: Per `mes` only. No `dia_semana` breakout in the fleet reference for this first slice.
2. **Benchmark visual treatment**: Percentile band overlaid on the existing evolution line chart. No radial gauge.
3. **`hora_salida` de-emphasis mechanism**: Full removal, not demotion. `_renderHoraChart` is deleted entirely (the dedicated always-on chart goes away completely, not collapsed/secondary).
4. **New CSV domain / path prefix**: Confirmed — `flota/` (e.g. `flota/percentiles_referencia.csv` + `flota/percentiles_empresa.csv`).
5. **Python script location**: Confirmed — directly in `Definitos/`, alongside its siblings (no subfolder).
6. **Period-comparison scope**: Confirmed (2026-07-14, after clarifying the grammatical ambiguity) — **two `mes` values, same company** for this first slice. No `dia_semana` crossing in period comparison.
7. **Consume the distribution (vs Ruido's dead columns)**: Yes — confirmed the reference file's p10..p90 columns must be actually rendered (band/markers), not parsed-and-ignored like Ruido's.
